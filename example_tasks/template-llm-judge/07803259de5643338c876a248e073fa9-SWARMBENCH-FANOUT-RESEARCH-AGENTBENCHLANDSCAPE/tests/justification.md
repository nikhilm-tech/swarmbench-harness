**num_domains = 2** — The paper constructs two domains: τ-retail and τ-airline. Table 1 states "Key statistics from τ-retail and τ-airline" and Section 4.1 describes: "we modularly construct two domains, τ-retail and τ-airline."

**num_tasks = 165** — Table 1 explicitly lists 115 tasks for τ-retail and 50 tasks for τ-airline, totaling 165 tasks.

**has_human_task_curation = true** — Section 4, Stage III describes manual task annotation: "we write an initial user instruction, run a trial with gpt-4-turbo function calling agent, polish the user instruction by examining the trajectory, and do this iteratively until we are certain no ambiguities exist." **has_refusal_ability = true** — The benchmark includes tasks where the agent must reject user requests that violate domain policies. The paper states: "As shown in Figure 2d, each τ-bench task instance has two parts: an instruction for the user simulation (hidden from agents), and an annotation of the ground truth database write actions (and optionally, ground truth outputs for user questions)." Additionally: "We can copy and edit agent actions and outputs for ground truth annotation, which is easier than annotating from scratch."

## Multi-Agent Failure Analysis

### Result: 8/11 benchmarks passed, match: false

### Fixes Applied Across Runs

**Run 1 → Run 2 fixes (task.json prompt + gold_decomposition):**
- benchmark_name: added "short name only, no subtitle" rule with examples → fixed ACEBench, WorkBench, TheAgentCompany, CRMArena title failures
- num_domains: clarified stakeholder personas/departments count as domains → fixed CRMArena regression
- num_domains: clarified pure cognitive/skill dimensions = "not provided" → fixed WorkArena++ regression
- has_refusal_ability: added "missing/erroneous parameter detection" as a refusal example → fixed ACEBench
- has_human_plans: added "checkpoint-based scoring without action sequences = false" → fixed TheAgentCompany regression
- synthesize-comparison: instructed synthesizer to always prefer evaluator-corrected values → fixed ITBench, WorkArena++
- Field definitions added to has_human_plans examples: Playwright oracles, API call chains, fault propagation chains → fixed ITBench

### Remaining Failures — Genuine Agent Limitations

**1. API-Bank num_tasks: agent extracted 2202, oracle is 314**
The API-Bank paper contains 2,202 total dialogues (1,888 training + 314 evaluation). The benchmark evaluation set is 314 manually annotated dialogues. The agent consistently extracts the total dataset size rather than the evaluation-only count. This requires careful reading of Table 2 to distinguish training from evaluation splits. Across 3 runs: "not provided" → 2202 → 2202. Genuine reading limitation.

**2. WorkArena num_tasks/curation/plans: inconsistent across runs**
WorkArena failures are non-deterministic — different fields fail each run (num_tasks=29 in some runs, has_human_task_curation=false in others, has_human_plans=false in others). The paper clearly states 33 tasks and hand-crafted oracle functions. This represents genuine LLM non-determinism in web retrieval and paper reading — different paper sections are read each run.

**3. CRMArena num_tasks: agent extracted 1170, oracle is 9**
CRMArena has 9 distinct task types but 1,170 query instances (130 per task). The agent consistently extracts the instance count rather than the task-type count. The paper describes both numbers prominently, requiring the agent to distinguish "task types" from "query instances." Across runs: consistently 1170. Genuine disambiguation limitation.
